# CLAUDE.md — Scientific Project Conventions

Instructions for Claude Code when working in this scientific repository. Follow these rules unless the user explicitly overrides them.

## Workflow rules

### Always plan before executing
- For any non-trivial task, enter plan mode first. Produce a written plan and wait for confirmation before writing code.
- Ask qualifying questions when requirements are ambiguous. Do not guess at intent.
- If the task involves data processing, state your assumptions about data shape, dtypes, units, and provenance before touching code.

### Stop and check in
- Do not run autonomously through long task chains. Pause at meaningful decision points.
- If you find yourself debugging the same issue for more than ~3 attempts without progress, stop and surface the problem rather than burning more context.
- After completing a discrete sub-task, summarise what changed before moving on.

### Test-driven development is the default
- For any new analysis function or data transformation, write tests first. The test defines correctness; the implementation satisfies it.
- Before refactoring, ensure tests cover the existing behaviour. Run tests after the refactor.
- For numerical/scientific code, include tests on synthetic data with known ground truth (e.g. known means, known parameters of a generative process).

### Use git aggressively
- Work on feature branches, not `main`/`master`.
- Commit at logical checkpoints with descriptive messages. Never bundle unrelated changes.
- Delete dead code with confidence — git is the backup. Do not leave commented-out code blocks.
- Before any large refactor or destructive change, confirm the working tree is clean.

## Repository structure

Default layout (true-neutral cookiecutter style):

```
data/
  raw/         # Immutable source data. Never write here.
  processed/   # Outputs of preprocessing pipelines.
  generated/   # Anything produced by an LLM/AI tool.
src/           # Importable package code (preprocessing, models, utilities).
notebooks/     # Exploratory analysis and visualisation only.
tests/         # Pytest tests for code in src/.
scripts/       # Entry-point scripts invoked by the orchestrator.
```

- `data/raw/` is read-only. If you need to modify raw data, copy it to `data/processed/` first.
- Anything generated by AI (synthetic datasets, mocked outputs, scaffolded files) goes under `data/generated/` so it is never confused with real experimental data.
- Package logic lives under `src/`. Notebooks import from `src/`; they do not duplicate logic.

## Separation of concerns

- **Data processing** belongs in pure Python/R modules under `src/`. It must be deterministic, testable, and importable.
- **Visualisation** belongs in notebooks or dedicated plotting scripts. Plots iterate frequently; processing should not.
- Never mix heavy data processing and plotting in the same file. If a notebook is doing significant computation, extract it into `src/` and import it.

## Dependency management

- Python: use `uv` for project virtual environments. System-wide tools via `mamba`.
- R: use `renv` for project dependencies.
- Pin versions in lockfiles. Do not silently update dependencies during unrelated tasks.
- If you add a new dependency, add it to the project file (`pyproject.toml`, `environment.yml`, etc.) — never just `pip install` into the env without recording it.

## Orchestration

- Pipelines must be reproducible end-to-end. There is a single command that runs the whole thing (Snakemake, Make, or a top-level shell script — check the repo to see which).
- New scripts must be wired into the orchestrator with explicit inputs and outputs. No hidden side effects, no hardcoded absolute paths.
- If you cannot tell where a piece of data came from, stop and trace its provenance before using it.

## Notebooks

Default policy: **prefer text-based notebooks** (marimo, jupytext, quarto) over `.ipynb`.

- If `.ipynb` files exist: when modifying, restart the kernel and run from scratch to verify the notebook is reproducible top-to-bottom. Do not commit notebooks with stale cell outputs that disagree with the code.
- If using marimo: respect the DAG structure — no hidden state across cells.
- For one-off analysis scripts that produce plots, prefer dumping `.png` files to a `figures/` directory over inline notebook plots.

## Visualisation strategy

- Generate **many cheap plots** when validating data. Sanity-check distributions, sample sizes, missingness, and units before any modelling.
- Default to static `matplotlib` / `seaborn` for paper-bound figures. Use `plotly`, `marimo`, or `streamlit` for interactive exploration.
- Every plot must have axis labels with units, a title, and a legend if multiple series are shown. No exceptions for "quick" plots — quick plots are exactly the ones that mislead.
- Save raw data alongside generated figures so plots can be regenerated without re-running the pipeline.

## Code hygiene

- Run `ruff` (Python) or equivalent linter before considering a task done.
- Name variables and functions to reflect what they represent in the science, not just the data type. `firing_rate_hz` not `arr2`.
- When the function of a piece of code changes, rename it. Stale names are worse than no names.
- Identify and remove dead code proactively. If unsure whether something is used, grep for it across the repo before deleting.

## Context and memory management

- When a task spans many turns, periodically write a handoff summary to a `NOTES.md` or task-specific markdown file so the work survives a context reset.
- When you solve a non-obvious bug, append a one-line entry to a `GOTCHAS.md` (or this file) so the same bug is not reintroduced later.
- Use `/clear` and `/compact` between unrelated tasks rather than letting the context drift.

## Things to flag, not fix silently

Stop and ask before doing any of the following:
- Modifying anything under `data/raw/`.
- Deleting or rewriting code you did not author in this session, beyond trivial cleanup.
- Changing the orchestrator pipeline structure or build system.
- Adding a heavyweight new dependency (deep learning frameworks, new languages, GPU libraries).
- Touching git history beyond the current branch (force-pushes, rebases of shared branches, history rewrites).

## Scientific correctness checks

Before claiming a result:
- Verify the data shape and units match what the analysis assumes.
- Check for off-by-one errors in time alignment, trial indexing, and behavioural epoch boundaries.
- Cross-validate any model fit. Report cross-validated metrics, not training-set fits.
- For any statistical claim, state the test, the assumptions, and whether they are met.
- If a result looks too clean or too good, treat that as a red flag and investigate before celebrating.
